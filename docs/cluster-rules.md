# Cluster Scriptの最低限のルール

Jcreate++を使う前に知っておくと迷いにくい、Cluster Scriptまわりの基本だけを短くまとめます。

ここでは初心者向けにざっくり説明します。細かい仕様はCluster公式ドキュメントも確認してください。

## Scriptable Item

Cluster Scriptを動かすには、基本的にスクリプトを実行するアイテムが必要です。

Unity側で対象アイテムに Scriptable Item を設定し、そのアイテムに生成したスクリプトを紐づけます。

よくある詰まりどころ:

- Scriptable Itemを付け忘れている
- スクリプトを設定したアイテムと、動かしたいアイテムが違う
- アイテム参照の名前がJcreate++側の指定と合っていない

## onStart

`onStart` は、スクリプトが動き始めたときの初期化に使う考え方です。

たとえば、最初の状態を保存する、カウントを0にする、表示状態を整える、といった処理に向いています。

## onUpdate

`onUpdate` は、時間経過に合わせて毎フレーム近い頻度で実行される処理です。

クールダウンの残り時間を減らす、なめらかに移動する、一定時間待つ、といった処理に使います。

注意点:

- 何でも `onUpdate` に入れると処理が重くなります。
- Jcreate++では、時間を使う処理をできるだけ1つの更新処理にまとめる方針です。
- 毎フレーム処理が必要ないものは、イベントで動かす方が見通しがよくなります。

## onInteract

`onInteract` は、プレイヤーがアイテムを触ったときに動く処理です。

ボタン、スイッチ、宝箱、説明パネルなど、「触ったら何かする」ギミックの入口になります。

よくある詰まりどころ:

- アイテムがInteract対象になっていない
- プレイヤーが触れる位置にColliderなどがない
- 触った対象とスクリプトが付いている対象を混同している

## $.state

`$.state` は、スクリプトの状態を覚えておくための場所として考えるとわかりやすいです。

例:

- ボタンが押されたか
- ドアが開いているか
- クールダウンが何秒残っているか
- ランダム判定の結果

注意点:

- 名前を間違えると別の状態として扱われます。
- 同じ名前を別の意味で使うと、思わぬ動きになります。
- 最初の値を決めておくと安全です。

## player / item

Cluster Scriptでは、処理の対象として「プレイヤー」や「アイテム」が出てきます。

- player: 触った人、近くにいる人、メッセージを送った人など
- item: スクリプトが付いているアイテム、参照している別アイテムなど

初心者のうちは、「誰に対して」「どのアイテムに対して」動かすのかをメモしてからブロックを組むと混乱しにくいです。

## 初心者が詰まること

- Unity側の設定が足りない
- Jcreate++側のブロックは正しいが、参照名が違う
- イベントが起きていない
- 状態の名前がずれている
- ランダムや条件分岐の結果を確認する手段がない
- 一度に大きな仕掛けを作りすぎて、どこで失敗したかわからない

まずは小さなギミックを1つ作り、動いたら少しずつ足していくのがおすすめです。

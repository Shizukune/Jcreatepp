# Jcreate++ Control Model Notes

## Purpose

ここまでの実装と議論を踏まえて、Jcreate++ の制御モデルを整理する。

特に以下を明確にする。

- `onUpdate` で起きる同時処理の扱い
- `if` の役割
- `sequence` の役割
- 順次処理を今後どう考えるか
- ブロック数が増えてきた問題への整理方針

## Current Understanding

画像のように `毎フレーム` の中へ複数の動作を並べた場合、生成コード上は `$.onUpdate` の中で上から順番に実行される。

ただし、毎フレーム同じタイミングで繰り返し実行されるため、見た目としては同時処理に近い。

例:

```txt
毎フレーム
  左右(X)方向に 幅:1 速さ:2 で往復する
  角度を x:0 y:0 z:0 ずつ変える
```

生成イメージ:

```js
$.onUpdate((deltaTime) => {
  // position update
  $.setPosition(...);

  // rotation update
  $.setRotation(...);
});
```

この場合:

- 往復移動は主に `$.setPosition(...)`
- 回転は主に `$.setRotation(...)`

なので、位置と回転を別々に更新しており、基本的には競合しない。

## Important Caveat

`onUpdate` 内のすべての処理が安全に合成されるわけではない。

例えば、同じフレーム内で複数の位置更新を行うと、意図しない干渉が起きる可能性がある。

例:

```txt
毎フレーム
  位置を x:0 y:0 z:0 にする
  左右(X)方向に 幅:1 速さ:2 で往復する
```

この場合、どちらも位置を更新するため、順番や上書きの影響を受ける。

現在問題ないと見ているのは、あくまで次のような組み合わせ。

```txt
位置を更新する継続処理 + 回転を更新する継続処理
```

## Conceptual Separation

現時点では、処理モデルを次のように分けるのが自然。

```txt
毎フレーム
  = 継続処理 / 同時っぽく見える処理

sequence
  = 開始されたあと、順番に処理を進める仕組み

if
  = 条件を見て、処理や sequence を開始する仕組み
```

## onUpdate

`onUpdate` は「常に起き続ける動作」に向いている。

例:

- 左右に往復する
- 回転し続ける
- 上下に浮く
- 追従する
- 毎フレーム条件を監視する

見た目としては、複数の継続動作が同時に起きる。

ただし、同じ種類の状態を複数ブロックが更新する場合は注意が必要。

分類:

```txt
位置系: setPosition / move / oscillate
回転系: setRotation / rotate
物理系: force
状態系: flag
```

同じ分類を複数同時に使うと、将来的には警告や設計上の整理が必要になる可能性がある。

## If

`if` は「条件を見て分岐する」ためのもの。

役割:

- 条件が true のときだけ処理する
- フラグによって動作を切り替える
- メッセージ受信後に条件で処理を変える
- sequence を開始するかどうかを決める

例:

```txt
メッセージ "open" を受け取ったとき
  もし フラグ doorEnabled なら
    一連の動作
      1 秒待つ
      位置を変える
```

`if` 自体は時間を進める仕組みではない。

時間をまたぐ処理は `sequence` 側で扱う。

## Sequence

`sequence` は「順次処理」のための仕組み。

役割:

- 上から順番に処理を進める
- `秒待つ` ができる
- `条件まで待つ` ができる
- イベントや `if` から開始される
- 実際の進行は `onUpdate` のステートマシンで行う

現状の意味:

```txt
sequence
  = 一度開始されると、内部ステップを onUpdate で少しずつ進めるタスク
```

今は主に以下ができる。

- 即時動作
- 秒待つ
- 条件まで待つ
- 次のステップに進む

## Sequential Behavior Still Needs Design

「まず左右に動く、次に回転する」のような順次演出は、今後もう少し設計が必要。

理由:

- 現在の `位置をずつ変える` は基本的に1回だけ実行される命令。
- 「2秒かけて動く」「1秒かけて回る」のような時間つき動作は、まだ専用設計が必要。

将来的な候補:

```txt
指定秒数かけて x:y:z ずつ動く
指定秒数かけて x:y:z まで動く
指定秒数かけて x:y:z ずつ回る
指定秒数かけて x:y:z まで回る
```

ただし、このまま単純に増やすとブロック数が急増する。

## Block Count Problem

現在すでにブロックが増えてきている。

このまま機能ごとに専用ブロックを増やすと、ユーザーが探しにくくなる。

問題:

- 似たブロックが増える
- 初心者がどれを使うべきかわからない
- ツールボックスが長くなる
- 「即時動作」「継続動作」「順次動作」の違いが見た目でわかりにくくなる

## Possible Organization Strategy

ブロックを機能名だけで並べるのではなく、処理モデルごとに分ける。

候補:

```txt
イベント
  開始時
  毎フレーム
  インタラクト時
  メッセージ受信

条件
  もし
  フラグ
  比較
  かつ / または / ではない

即時動作
  位置を変える
  角度を変える
  フラグを変える

継続動作
  往復する
  回転し続ける

順次動作
  一連の動作
  秒待つ
  条件まで待つ
  時間をかけて動く
  時間をかけて回る

完成済みギミック
  乗り物
```

この分け方だと、ユーザーに以下の区別を伝えやすい。

- すぐ実行する
- ずっと実行する
- 順番に実行する

## Possible Block Design Direction

ブロック数を増やしすぎないために、動作ブロックを「モード選択型」にする案がある。

例:

```txt
位置を [すぐに / 3秒かけて] x:y:z [にする / ずつ変える]
```

または:

```txt
角度を [すぐに / 3秒かけて] x:y:z [にする / ずつ変える]
```

メリット:

- ブロック数を減らせる
- 位置系・回転系の概念がまとまる
- 即時処理と順次処理の差をUI上で表現できる

デメリット:

- 1ブロックの見た目が複雑になる
- Blockly上で入力項目が増えすぎる可能性がある

MVPでは、無理に統合せず、まず分類を明確にするだけでもよい。

## Recommended Mental Model

現時点でのおすすめの整理:

```txt
毎フレーム
  同時に起き続ける動作用

sequence
  順番に起きる演出用

if
  どの動作・演出を起動するか決める条件用
```

このモデルなら、以下のように説明できる。

```txt
左右に動きながら回転する
  -> 毎フレーム

まず動く、次に回転する
  -> sequence

フラグがtrueなら動く
  -> if
```

## Current Conclusion

現状の `毎フレーム` 内の複数ブロックは、見た目として同時処理になる。

画像のような「往復しながら回転」は、位置更新と回転更新が別なので基本的に問題ない。

一方で、「まず動く、次に回転する」のような順次処理は、別の概念として `sequence` で扱う。

今後の課題は、順次処理用の動作を追加するときに、ブロック数を増やしすぎない設計にすること。
